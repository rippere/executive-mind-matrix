## Implementation Details

### Endpoints Implemented

#### 1. `POST /webhooks/slack`
- Receives Slack slash commands
- HMAC SHA256 signature verification
- Replay attack prevention (5-minute window)
- Returns immediate response to Slack (required)
- Creates System Inbox entry with Slack metadata

**Usage**: `/intent <text>` in Slack

#### 2. `POST /webhooks/email`
- Receives forwarded emails from Zapier/Make
- Parses subject and body
- Stores sender information as metadata
- No authentication required (use Zapier/Make's security)

**Payload**:
```json
{
  "subject": "Email subject",
  "body": "Email content",
  "from_email": "sender@example.com",
  "from_name": "Sender Name",
  "received_date": "2024-01-01T12:00:00Z"
}
```

#### 3. `POST /webhooks/generic`
- Generic webhook for any external trigger
- API key authentication via `X-API-Key` header
- Flexible metadata support
- Priority levels: low, medium, high

**Payload**:
```json
{
  "title": "Intent title",
  "content": "Intent description",
  "priority": "high",
  "source_system": "Custom System",
  "metadata": {}
}
```

#### 4. `GET /webhooks/health`
- Health check endpoint
- Returns configuration status
- Shows which security features are enabled

## Security Features

### Implemented Security Measures

1. **Slack HMAC Signature Verification**
   - Uses `SLACK_SIGNING_SECRET` for HMAC SHA256
   - Prevents unauthorized requests
   - Constant-time comparison (prevents timing attacks)
   - Replay attack prevention (5-minute window)

2. **API Key Authentication (Generic Webhook)**
   - Uses `WEBHOOK_API_KEY` for authentication
   - Constant-time comparison
   - Header-based authentication (`X-API-Key`)

3. **Rate Limiting**
   - Inherits from application-wide rate limiting
   - Default: 60 requests per minute per IP
   - Configurable via `RATE_LIMIT_PER_MINUTE`

4. **Input Validation**
   - Pydantic models for all payloads
   - Field length limits
   - Type validation
   - Custom validators (priority levels, etc.)

5. **Error Handling**
   - Comprehensive try-catch blocks
   - Structured error responses
   - Detailed logging
   - Metrics tracking for errors

## Integration with Existing System

### Notion Integration

All webhooks create entries in the System Inbox database:

```python
properties = {
    "Content": {"rich_text": [{"text": {"content": content}}]},
    "Source": {"select": {"name": source}},  # "Slack", "Email", "Webhook"
    "Received_Date": {"date": {"start": datetime.now().date().isoformat()}},
    "Status": {"select": {"name": "Unprocessed"}},
    "Source_Metadata": {"rich_text": [{"text": {"content": metadata}}]}
}
```

### Polling Integration

- Webhook creates entry with `Status: Unprocessed`
- Existing poller picks it up in next cycle (2 minutes)
- Intent is classified and routed automatically
- No changes needed to existing polling logic

### Monitoring Integration

Prometheus metrics automatically tracked:
- `items_processed_total{type="webhook_slack"}`
- `items_processed_total{type="webhook_email"}`
- `items_processed_total{type="webhook_webhook"}`
- `errors_total{type="*",component="webhook*"}`

   - Generic: `https://your-app.railway.app/webhooks/generic`

### Security Considerations

1. Always use HTTPS in production (Railway provides this)
2. Rotate API keys regularly
3. Monitor webhook logs for suspicious activity
4. Use Slack signature verification (don't skip it)
5. Set appropriate rate limits

## Troubleshooting

See `docs/WEBHOOK_SETUP.md` for detailed troubleshooting guide.

Common issues:

1. **"Webhooks are disabled"**
   - Check `ENABLE_WEBHOOKS=true` in `.env`
   - Restart application

2. **"Invalid signature" (Slack)**
   - Verify `SLACK_SIGNING_SECRET` is correct
   - Check server time is synchronized

3. **401 Unauthorized (Generic)**
   - Verify `X-API-Key` header is included
   - Check API key matches `WEBHOOK_API_KEY`

4. **Intent not appearing**
   - Check webhook response was successful
   - Wait for next poll cycle (up to 2 minutes)
   - Check application logs
